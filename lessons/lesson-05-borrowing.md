# Урок 5. Ссылки и заимствования (References & Borrowing)

В прошлом уроке мы научились отдавать владение и получать его обратно. Делать так на каждой функции — неудобно. Поэтому в Rust есть **ссылки** — способ дать функции временный доступ к данным без передачи владения. Borrow-checker гарантирует, что эти ссылки всегда валидны.

## Теория

### Зачем нужны ссылки

```rust
fn calculate_length(s: String) -> (String, usize) {
    let len = s.len();
    (s, len)         // приходится возвращать строку, чтобы не потерять владение
}
```

Со ссылками то же самое выглядит так:

```rust
fn calculate_length(s: &String) -> usize {
    s.len()
}

fn main() {
    let s = String::from("hello");
    let len = calculate_length(&s);
    println!("длина {s} = {len}");
}
```

`&s` — это **ссылка** на `s`: она даёт временный, не-владеющий доступ к данным.

### Иммутабельные и мутабельные ссылки

Есть два вида ссылок:

- `&T` — иммутабельная ссылка, чтение.
- `&mut T` — мутабельная ссылка, чтение и запись.

```rust
fn append(s: &mut String) {
    s.push_str(" world");
}

fn main() {
    let mut s = String::from("hello");
    append(&mut s);
    println!("{s}");
}
```

Чтобы взять `&mut`, переменная сама должна быть `mut`.

### Правила заимствования

Это центральные правила borrow-checker'а:

В любой момент времени для одних данных можно иметь либо:

- любое число `&T` (иммутабельных ссылок),
- **либо** ровно одну `&mut T` (мутабельную),
- но **не одновременно**.

И ещё: ссылки **всегда валидны** — borrow-checker не пропустит ситуацию, в которой ссылка переживёт владельца.

```rust
fn main() {
    let mut s = String::from("hi");
    let r1 = &s;
    let r2 = &s;
    println!("{r1} {r2}");      // оба читателя живы — ок
    let m = &mut s;             // ок: r1, r2 больше не используются
    m.push_str("!");
    println!("{m}");
}
```

Зачем такие правила? Они устраняют гонки данных и инвалидацию ссылок при модификации коллекций. Это аналогично паттерну «один писатель ИЛИ много читателей» из конкурентного программирования, только проверяется компилятором.

### NLL (Non-Lexical Lifetimes)

Раньше время заимствования определялось блоками `{ ... }`, и многие легитимные программы не компилировались. Сейчас borrow-checker умнее и считает время жизни ссылок по их **фактическому использованию**:

```rust
let mut v = vec![1, 2, 3];
let r = &v[0];
println!("{r}");        // здесь r последний раз нужен
v.push(4);              // ок: после println! r не используется
println!("{:?}", v);
```

Без NLL пришлось бы оборачивать в блок. С NLL компилятор сам видит, что после `println!` ссылка не нужна.

### Висячие ссылки запрещены

```rust
fn dangle() -> &String {       // ошибка: missing lifetime specifier
    let s = String::from("x");
    &s                          // ссылка на локальную переменную
}
```

Внутри функции `s` дропается в конце, и ссылка на неё стала бы висячей. Компилятор такое не пропустит.

Решение — вернуть значение по владению:

```rust
fn no_dangle() -> String {
    String::from("x")
}
```

### Slice-ссылки

Ссылка на часть массива или строки — это `&[T]` и `&str`. Они состоят из указателя и длины, и тоже подчиняются правилам заимствования:

```rust
let v = vec![10, 20, 30, 40];
let middle: &[i32] = &v[1..3];
println!("{middle:?}");        // [20, 30]

let s = String::from("hello world");
let hello: &str = &s[..5];
println!("{hello}");
```

Подробнее в уроке 6.

### Lifetimes (краткое введение)

Каждая ссылка имеет **время жизни** (lifetime). Чаще всего его выводит компилятор по правилам elision, но иногда нужно указать явно:

```rust
fn longest<'a>(a: &'a str, b: &'a str) -> &'a str {
    if a.len() > b.len() { a } else { b }
}
```

`'a` — это параметр времени жизни. Запись `-> &'a str` говорит: «возвращаемая ссылка живёт не дольше, чем меньшая из `a` и `b`». Подробно — в уроке 9.

Правила elision (упрощённо):

1. Каждый аргумент-ссылка получает свой `'a`.
2. Если есть только один входной lifetime, он автоматически становится lifetime результата.
3. Если есть `&self` или `&mut self`, lifetime результата привязывается к `self`.

Поэтому многие сигнатуры пишут без `'a` — компилятор сам всё расставляет.

### Reborrow

Когда вы передаёте `&mut T` в функцию, кажется, что владение ссылкой уходит. На самом деле компилятор делает **reborrow**: создаёт новую ссылку `&mut *r` с более коротким временем жизни, передаёт её, а оригинал «приостанавливается» и снова становится доступен после возврата:

```rust
fn poke(r: &mut i32) { *r += 1; }

fn main() {
    let mut x = 10;
    let r = &mut x;
    poke(r);    // на самом деле передаётся reborrow &mut *r
    poke(r);    // r снова доступен, снова reborrow
    println!("{r}");
}
```

Это объясняет, почему `&mut T` ведёт себя «как move», но при этом одну и ту же ссылку можно использовать после вызова функции.

### Деревья заимствований и поля структур

Borrow-checker умеет отслеживать заимствования отдельных полей структуры:

```rust
struct P { x: i32, y: i32 }

let mut p = P { x: 1, y: 2 };
let rx = &mut p.x;
let ry = &mut p.y;     // ок: разные поля
*rx += 10;
*ry += 20;
println!("{} {}", p.x, p.y);
```

Но, например, `&mut p.x` и `&mut p` одновременно — нельзя, потому что `&mut p` подразумевает доступ ко всем полям.

### Распространённые ошибки

1. **cannot borrow as mutable because it is also borrowed as immutable** — попытка взять `&mut`, пока живы `&`. Решение: завершить использование иммутабельных раньше или вынести в отдельный блок.
2. **cannot borrow as immutable because it is also borrowed as mutable** — обратная ситуация.
3. **does not live long enough** — ссылка переживает данные, на которые ссылается.
4. **lifetime mismatch** — компилятор не может вывести совместимые времена жизни. Решение: добавить параметры `'a`.

## Практика

### Задача 1. Длина строки через ссылку

```rust
fn length(s: &str) -> usize { s.len() }

fn main() {
    let s = String::from("привет");
    println!("байт: {}", length(&s));
    println!("байт: {}", length("hello"));
}
```

Принимая `&str`, функция работает и со `String` (через автоприведение `&String -> &str`), и со строковыми литералами.

### Задача 2. Изменение через &mut

```rust
fn add_exclamation(s: &mut String) {
    s.push('!');
}

fn main() {
    let mut greeting = String::from("привет");
    add_exclamation(&mut greeting);
    add_exclamation(&mut greeting);
    add_exclamation(&mut greeting);
    println!("{greeting}");   // привет!!!
}
```

### Задача 3. Ссылка на самое длинное слово

```rust
fn longest<'a>(a: &'a str, b: &'a str) -> &'a str {
    if a.len() > b.len() { a } else { b }
}

fn main() {
    let s1 = String::from("короткое");
    let s2 = String::from("очень длинное");
    println!("{}", longest(&s1, &s2));
}
```

### Задача 4. Несколько читателей и один писатель

```rust
fn main() {
    let mut v = vec![1, 2, 3];

    let r1 = &v;
    let r2 = &v;
    println!("сумма читателей: {}", r1.iter().sum::<i32>() + r2.iter().sum::<i32>());
    // r1 и r2 больше не используются — NLL

    let r3 = &mut v;
    r3.push(4);
    println!("после push: {:?}", r3);
}
```

### Задача 5. Свап через ссылки

```rust
fn swap(a: &mut i32, b: &mut i32) {
    std::mem::swap(a, b);
}

fn main() {
    let mut x = 1;
    let mut y = 2;
    swap(&mut x, &mut y);
    println!("{x} {y}");      // 2 1
}
```

`std::mem::swap` принимает две `&mut` и обменивает их содержимое без аллокаций.

### Задача 6. Подсчёт символов через ссылку на &str

```rust
fn count_chars(s: &str) -> (usize, usize) {
    (s.len(), s.chars().count())
}

fn main() {
    for s in ["ru", "русский", "🦀🦀"] {
        let (b, c) = count_chars(s);
        println!("{s} -> bytes={b}, chars={c}");
    }
}
```

### Задача 7. Распространённая ошибка с iter mut

```rust
fn main() {
    let mut v = vec![1, 2, 3, 4];
    for x in v.iter_mut() {
        *x *= 2;
    }
    println!("{v:?}");      // [2, 4, 6, 8]
}
```

`iter_mut` даёт мутабельные ссылки на каждый элемент. Без `*x` мы бы пытались присвоить `&mut i32` целое — компилятор не разрешит.

### Задача 8. Reborrow на практике

```rust
fn inc(r: &mut i32) { *r += 1; }
fn dec(r: &mut i32) { *r -= 1; }

fn main() {
    let mut x = 5;
    let r = &mut x;
    inc(r);
    dec(r);
    inc(r);
    println!("{r}");        // 6
}
```

Компилятор делает reborrow при каждом вызове, так что одну и ту же ссылку `r` можно передавать многократно.

**10.** Почему `&mut T` не реализует `Copy`?

---

### Ответы

1. `r` — это `&i32` внутрь буфера вектора. `push` берёт `&mut self` и может реаллоцировать буфер, после чего `r` указывал бы в освобождённую память. Borrow-checker запрещает `&mut`-вызов, пока живы `&`-ссылки.

2. Не компилируется. `r1` используется после создания `&mut`, значит времена жизни иммутабельной и мутабельной ссылок пересекаются. Если поменять порядок `println!` (сначала `r1`, потом `r2`), будет ок благодаря NLL.

3. Вернёт подсрез входной строки. Lifetime выводится по правилу elision: единственный входной lifetime автоматически привязывается к выходному. Это эквивалентно `fn first_word<'a>(s: &'a str) -> &'a str`.

4. Да. `&str` — это жирный указатель (ptr + len), без владения данными. Копирование двух машинных слов безопасно.

5. `&mut T` не реализует `Copy`, поэтому передача в функцию похожа на move. Но компилятор автоматически делает reborrow: создаёт новую ссылку `&mut *r` с более коротким lifetime, оригинал «приостанавливается» на время вызова и снова становится доступен после.

6. Напечатает `hello hello` и потом `hello`. Несколько `&`-ссылок на один `&`-источник — это нормально и в любом количестве.

7. Не скомпилируется: компилятор не может вывести lifetime результата (есть два входа с разным временем жизни). Чинится явным lifetime: `fn longest<'a>(a: &'a str, b: &'a str) -> &'a str`.

8. `&String` принимает только ссылки на `String`. `&str` принимает и `String` (через автоприведение `&String -> &str`), и строковые литералы, и подсрезы. Идиоматично всегда писать `&str` — это более общий и удобный API.

9. Напечатает `[1, 2, 3]` и `[1, 2, 3, 4]`. Иммутабельная ссылка `r` живёт только внутри блока, после блока её нет, поэтому `v.push(4)` корректен.

10. Если бы `&mut T` был `Copy`, то можно было бы получить два мутабельных алиаса на одни данные. Это нарушило бы фундаментальный инвариант Rust «ровно один `&mut` или сколько угодно `&`», что открыло бы дверь к гонкам данных и UB. Поэтому `&mut T` намеренно не `Copy` и не `Clone`.
# Урок 5. Ссылки и заимствования (References & Borrowing)

## Теория

Ссылка `&T` даёт доступ к значению без передачи владения. Ссылки **всегда валидны** — это инвариант borrow-checker'а.

### Правила заимствования

В любой момент времени можно иметь или:

- любое число `&T` (неизменяемых ссылок),
- **либо** ровно одну `&mut T` (изменяемую),
- но не одновременно.

```rust
fn main() {
    let mut s = String::from("hi");
    let r1 = &s;
    let r2 = &s;
    println!("{r1} {r2}");
    let m = &mut s;   // ок: r1, r2 больше не используются
    m.push_str("!");
}
```

### NLL (Non-Lexical Lifetimes)

Время заимствования определяется не блоком, а фактическим использованием. Компилятор видит, что после `println!` ссылки рабочей больше не нужны.

### Dangling references

```rust
fn dangle() -> &String {     // ошибка
    let s = String::from("x");
    &s
}
```

Ссылка бы указывала на дропнувшую строку. Компилятор отловит это.

## Практика

```rust
fn longest<'a>(a: &'a str, b: &'a str) -> &'a str {
    if a.len() > b.len() { a } else { b }
}

fn add_exclamation(s: &mut String) {
    s.push('!');
}

fn main() {
    let mut greeting = String::from("hello");
    add_exclamation(&mut greeting);
    let other = String::from("hi");
    println!("{}", longest(&greeting, &other));
}
```

    let r = &v[0];

    println!("{r}");
}
```

fn main() {
    let mut s = String::from("x");
    let r1 = &s;
    let r2 = &mut s;
    println!("{r2}");
    println!("{r1}");

```

**3.** Что вернёт функция?
```rust

    s.split_whitespace().next().unwrap_or("")

```
Почему здесь не нужно явно указывать lifetime?

**4.** `&str` всегда реализует `Copy`? Почему?

**5.** Чем reborrow отличается от move для `&mut T`?

---

### Ответы

1. `r` — `&i32` внутрь буфера. `push` может реаллоцировать, и `r` будет указывать в освобождённую память. Borrow-checker запрещает `&mut`-вызов при живой `&`.

2. Не компилируется. `r1` используется после создания `&mut`, значит время жизни обеих пересекается. Переставьте вывод `r1` до `&mut` — и будет ок (NLL).

3. Вернёт подсрез входной строки. Lifetime выводится по правилу elision: единственный input lifetime автоматически привязывается к output.

4. Да. `&str` — фактически жирный указатель (ptr+len), без владения данными. Копирование безопасно.

5. `&mut T` не реализует `Copy`, поэтому передача в функцию похожа на move. Но компилятор автоматически делает reborrow `&mut *r`: оригинал приостанавливается на время вызова и возвращается после.
