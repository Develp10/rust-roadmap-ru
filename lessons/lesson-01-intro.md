# Урок 1. Введение в Rust, установка и первая программа

Этот урок — фундамент. Дальше всё строится на привычке гонять `cargo`, понимать разницу между `build` и `check`, и знать, какие компоненты входят в тулчейн. Если вы уверенно настроили окружение и запустили первую программу, дальше будет проще.

## Теория

### Что такое Rust и почему его выбирают

Rust — это системный язык программирования, спроектированный с тремя главными целями:

1. **Безопасность памяти без сборщика мусора.** Никаких `null`-указателей, висячих ссылок, гонок данных, double-free и use-after-free — это всё ловится на этапе компиляции.
2. **Производительность уровня C/C++.** Без рантайма, с zero-cost abstractions: дженерики, итераторы, замыкания компилируются в максимально эффективный код.
3. **Эргономика и инструментарий.** Современный пакетный менеджер, форматтер, линтер, документация, тестирование и LSP-сервер — всё «из коробки».

В индустрии на Rust пишут операционные системы (часть Linux, Redox, Fuchsia), браузерные движки (Servo, части Firefox), сетевые сервисы (Cloudflare, Discord, Dropbox), embedded и IoT (микроконтроллеры, RTIC), блокчейн (Solana, Polkadot), геймдев (Bevy), CLI-инструменты (`ripgrep`, `fd`, `bat`, `exa`) и ML-инфраструктуру (Hugging Face Candle, тяжёлые ядра PyTorch).

Безопасность достигается через систему **владения** (ownership), **заимствований** (borrowing) и **времён жизни** (lifetimes), которая проверяется компилятором. Платой за это является более крутая кривая входа — компилятор будет «спорить» с вами на старте, но он же избавит от целого класса багов в проде.

### Установка тулчейна через rustup

Официальный установщик называется `rustup`. Он управляет:

- **Тулчейнами** — `stable`, `beta`, `nightly`. По умолчанию используется `stable`, на nightly включаются экспериментальные фичи.
- **Таргетами** кросс-компиляции — например, `wasm32-unknown-unknown`, `aarch64-unknown-linux-gnu`.
- **Компонентами**: `rustfmt` (форматтер), `clippy` (линтер), `rust-analyzer` (LSP), `rust-src` (исходники стандартной библиотеки), `miri` (интерпретатор для поиска UB).

Установка на macOS/Linux:

```bash
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
```

На Windows запускают `rustup-init.exe` с rustup.rs.

После установки полезно знать:

```bash
rustup update                       # обновить все тулчейны
rustup default stable               # сделать stable по умолчанию
rustup toolchain install nightly    # поставить nightly
rustup component add clippy rustfmt rust-analyzer
rustup target add wasm32-unknown-unknown
rustup show                         # что установлено и что активно
rustc --version
cargo --version
```

### Cargo: пакетный менеджер и сборщик

`cargo` — это «швейцарский нож» Rust. Минимальный набор команд, которые надо выучить:

| Команда | Что делает |
|---------|-----------|
| `cargo new <name>` | создаёт бинарный проект (`--lib` для библиотеки) |
| `cargo init` | инициализирует проект в текущей папке |
| `cargo build` | собирает в `target/debug` |
| `cargo build --release` | оптимизированная сборка в `target/release` |
| `cargo run` | собрать и запустить |
| `cargo run -- arg1 arg2` | запуск с аргументами |
| `cargo check` | быстрая проверка типов без кодогенерации |
| `cargo test` | прогон тестов |
| `cargo clippy` | линтер |
| `cargo fmt` | форматирование |
| `cargo doc --open` | документация |
| `cargo add <crate>` | добавить зависимость в Cargo.toml |
| `cargo tree` | дерево зависимостей |
| `cargo clean` | удалить `target` |

`Cargo.toml` — манифест проекта (метаданные, зависимости, edition). `Cargo.lock` — снимок точных версий, нужный для воспроизводимости.

### Анатомия проекта

После `cargo new hello_rust` появится:

```
hello_rust/
├── .git/
├── .gitignore
├── Cargo.toml
└── src/
    └── main.rs
```

`src/main.rs` — точка входа бинарного крейта. Для библиотеки это был бы `src/lib.rs`.

Содержимое `Cargo.toml`:

```toml
[package]
name = "hello_rust"
version = "0.1.0"
edition = "2021"

[dependencies]
```

`edition` — это «диалект» языка (2015, 2018, 2021, 2024). Он не ломает совместимость крейтов между собой, а только меняет некоторые правила парсера и стандартные импорты внутри одного крейта. По умолчанию выбирайте последнюю.

### Первая программа: hello, world

```rust
fn main() {
    println!("Hello, Rust!");
}
```

Запоминаем сразу несколько важных вещей:

- `fn main()` — точка входа исполняемого крейта. У `main` обычно тип `()` или `Result<(), E>`.
- `println!` — это **макрос**, а не функция. Восклицательный знак — маркер вызова макроса. Макрос проверяет форматную строку на этапе компиляции и поддерживает переменное число аргументов.
- Тело функции — это блок `{ ... }`, который сам по себе является выражением.
- В конце инструкции ставится `;`. Без `;` строка превращается в выражение-результат блока (это понадобится в следующих уроках).

### Форматирование вывода

Rust использует тот же стиль плейсхолдеров, что и Python:

```rust
fn main() {
    let name = "Алиса";
    let age = 30;
    println!("Привет, {}! Тебе {}.", name, age);     // позиционно
    println!("Привет, {name}! Тебе {age}.");          // именованно (с 1.58)
    println!("{0} и {1}, {0} снова", "alpha", "beta"); // по индексу
    println!("{:>10}", "right");      // выравнивание по правому краю в 10 колонок
    println!("{:0>5}", 42);            // 00042
    println!("{:.3}", 3.1415926);      // три знака после запятой
    println!("{:#?}", vec![1, 2, 3]); // pretty-print Debug
    eprintln!("это идёт в stderr");
}
```

### cargo build vs cargo check vs cargo run

Понимание разницы экономит часы:

- `cargo check` — проходит фронтенд компилятора: парсинг, разрешение имён, проверку типов, borrow checker. Не запускает кодогенерацию LLVM и линковку. Поэтому в 3-10 раз быстрее.
- `cargo build` — добавляет кодогенерацию (LLVM IR -> машинный код) и линковку. Получается артефакт.
- `cargo build --release` — оптимизирующая сборка. Медленнее, но в десятки раз быстрее по времени выполнения.
- `cargo run` = `cargo build` + запуск.

В цикле «правлю код — смотрю ошибки» используют `cargo check` или `cargo clippy`. К `build` обращаются, когда нужен исполняемый файл.

### Основные ловушки новичка

1. **Не путать `println!` и `print!`** — первый добавляет перевод строки.
2. **`println!("{}", x)` требует `Display`**, а `println!("{:?}", x)` — `Debug`. У большинства собственных типов первого нет, второй можно автоматически добавить через `#[derive(Debug)]`.
3. **`{name}` работает только для переменных в скоупе.** Если хотите подставить выражение, делайте `let tmp = ...; println!("{tmp}");` или `println!("{}", expr)`.
4. **Закройте все скобки.** Компилятор Rust очень помогает, читайте сообщения об ошибках до конца — там есть подсказки и предлагаемые исправления.

## Тест

**1.** В чём принципиальная разница между `cargo build` и `cargo check`, и почему `check` может быть в разы быстрее?

**2.** Должен ли `Cargo.lock` коммититься в репозиторий бинарного приложения? А библиотеки? Почему правила разные?

**3.** Скомпилируется ли такой `main`?

```rust
fn main() -> i32 {
    0
}
```

**4.** Что выведет программа?

```rust
fn main() {
    let x = { let x = 1; let x = x + 1; x * 10 };
    println!("{}", x);
}
```

**5.** Почему `println!` — это макрос, а не функция?

**6.** Чем отличаются `stable`, `beta` и `nightly` тулчейны и зачем нужен каждый?

**7.** Что напечатает программа и почему?

```rust
fn main() {
    let x = 7;
    println!("{x:>5}|{x:0>5}|{x:<5}|{x:^5}");
}
```

**8.** В чём разница между `cargo run` и `cargo run --release` для CPU-bound программы?

---

### Ответы

1. `cargo check` доходит только до фронтенда компилятора: парсинг, проверка типов, borrow checker. Кодогенерация LLVM и линковка — самые тяжёлые шаги `build` — пропускаются. Поэтому `check` особенно полезен в цикле разработки, когда нужно быстро узнать, валиден ли код типизационно.

2. Для бинаря — да, коммитят: `Cargo.lock` фиксирует версии зависимостей и обеспечивает воспроизводимость продакшен-сборок. Для библиотеки обычно не коммитят: пользователь библиотеки решает версии сам через свой `Cargo.lock`, иначе возникали бы конфликты в дереве зависимостей.

3. Не скомпилируется в обычном виде. `main` должен возвращать тип, реализующий трейт `Termination`. Прямо его реализуют `()`, `Result<(), E>` где `E: Debug`, `std::process::ExitCode` и `Infallible`. `i32` — нет.

4. Выведет `20`. Внутренний блок — выражение: первое `x = 1`, потом shadowing `x = 2`, последняя строка `x * 10` без `;` становится возвращаемым значением блока.

5. Макрос проверяет форматную строку на этапе компиляции (`println!("{}", x)` упадёт компиляцией, если у `x` нет `Display`), принимает переменное число аргументов разных типов и раскрывается в код. Обычные функции в Rust не поддерживают вариадику — поэтому `println!` реализован как макрос.

6. `stable` — релизная версия, обновляется раз в 6 недель, гарантия обратной совместимости. `beta` — будущий stable, попадает в релиз через 6 недель. `nightly` — ежедневный билд, в нём доступны экспериментальные фичи через `#![feature(...)]`. Для приложений и библиотек выбирают stable; nightly — для тех, кому нужны фичи (например, `miri`, ряд макросов) или кто работает над компилятором.

7. Напечатает `    7|00007|7    |  7  `. Спецификаторы выравнивания: `>` — по правому краю, `<` — по левому, `^` — по центру. Символ перед знаком (`0`) — это паддинг.

8. `cargo run` собирает в профиле `dev` без оптимизаций — компиляция быстрая, бинарник медленный. `cargo run --release` использует профиль `release` с `opt-level = 3`, инлайнингом и LLVM-оптимизациями: компиляция занимает в разы дольше, зато на CPU-bound коде разница в скорости легко 5-50x. Для замеров производительности всегда берут release.
# Урок 1. Введение в Rust, установка и первая программа

## Теория

Rust — это системный язык программирования с фокусом на безопасность памяти, производительность и параллелизм без сборщика мусора. Безопасность достигается через систему владения (ownership), заимствований (borrowing) и времён жизни (lifetimes), которая проверяется на этапе компиляции.

### Установка

Официальный установщик — `rustup`. Он управляет:
- тулчейнами (stable, beta, nightly);
- таргетами кросс-компиляции;
- компонентами: `rustfmt`, `clippy`, `rust-analyzer`, `rust-src`.

Команды:

```bash
rustup update
rustup default stable
rustup component add clippy rustfmt
rustc --version
cargo --version
```

### Cargo

`cargo` — менеджер пакетов и сборщик. Основные команды:
- `cargo new <name>` — создать проект (`--lib` для библиотеки);
- `cargo build` / `cargo build --release` — сборка;
- `cargo run` — собрать и запустить;
- `cargo check` — быстрая проверка без генерации машинного кода;
- `cargo test` — запустить тесты;
- `cargo clippy` — линтер;
- `cargo fmt` — форматирование.

`Cargo.toml` — манифест: метаданные, зависимости, edition.
`Cargo.lock` — зафиксированные версии зависимостей.

### Первая программа

```rust
fn main() {
    println!("Hello, Rust!");
}
```

`println!` — макрос (с восклицательным знаком), а не функция. `fn main()` — точка входа.

## Практика

1. Установите `rustup`, проверьте версии `rustc` и `cargo`.
2. Создайте проект: `cargo new hello_rust && cd hello_rust`.
3. Реализуйте функцию приветствия:

```rust
fn greet(name: &str) {
```

5. Запустите `cargo clippy -- -D warnings` — почините все замечания.

## Тест (хитрые вопросы)

**1.** В чём принципиальная разница между `cargo build` и `cargo check`, и почему `check` может быть в разы быстрее?

**2.** Должен ли `Cargo.lock` коммититься в репозиторий бинарного приложения? А библиотеки? Почему правила разные?

**3.** Скомпилируется ли такой `main`?

```rust
fn main() -> i32 {
    0
}
```

**4.** Что выведет программа?

```rust
fn main() {
```

**5.** Почему `println!` — это макрос, а не функция?

---

### Ответы

1. `cargo check` доходит только до фронтенда компилятора: парсинг, проверка типов, borrow-checker — но не запускает кодогенерацию LLVM и линковку. Это самая медленная часть `build`, поэтому `check` ускоряет цикл разработки.

2. Для бинаря — **да, коммитят**: важна воспроизводимость продакшен-сборок. Для библиотеки — **нет**: конечному пользователю должны достаться гибкие версии, иначе возникают конфликты в дереве зависимостей.

3. **Не скомпилируется** в обычном виде: `main` должен возвращать тип, реализующий трейт `Termination`. `i32` его напрямую не реализует — нужно `()`, `Result<(), E>` или `ExitCode`.

4. Выведет `20`. Внутренний блок — это выражение, последняя строка `x * 10` без `;` становится возвращаемым значением блока.

5. Макросы умеют принимать переменное число аргументов разных типов и проверять форматную строку на этапе компиляции. Обычные функции в Rust не поддерживают вариадику, поэтому `println!` реализован как макрос.
