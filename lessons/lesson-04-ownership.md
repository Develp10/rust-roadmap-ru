# Урок 4. Ownership (владение)

Ownership — это самая важная и самая «непривычная» особенность Rust. Именно она позволяет языку обходиться без сборщика мусора и при этом гарантировать отсутствие use-after-free, double-free, утечек по неосторожности и data races.

## Теория

### Зачем нужно владение

В большинстве языков управление памятью выглядит одним из двух способов:

1. **Сборщик мусора** (Java, Go, Python): среда сама находит «мусор» и освобождает. Просто, но добавляет рантайм и непредсказуемые паузы.
2. **Ручное управление** (C, C++): программист сам зовёт `free`/`delete`. Эффективно, но именно отсюда берутся 70% уязвимостей памяти в крупном софте.

Rust выбирает третий путь: **компилятор статически проверяет правила, по которым ресурс получает один владелец и освобождается, когда тот выходит из области видимости**. Никакой сборки мусора, никаких ручных `free` — и при этом нельзя обратиться к освобождённой памяти.

### Три правила ownership

1. У каждого значения есть **ровно один** владелец.
2. В каждый момент времени владелец один (ownership можно передавать, но не дублировать).
3. Когда владелец выходит из области видимости — значение **дропается** (вызывается `Drop`).

Эти три правила исключают целые классы багов. Применяются они на этапе компиляции, поэтому в рантайме никаких накладных расходов.

### Stack vs Heap

Чтобы понимать ownership, важно различать стек и кучу:

- **Стек** — быстрый, LIFO, размер каждого значения известен на этапе компиляции. Здесь живут `i32`, `bool`, `char`, ссылки, фиксированные массивы.
- **Куча** — для значений неизвестного или большого размера; выделение через аллокатор. Здесь живут данные `String`, `Vec`, `Box`, `HashMap`.

Когда мы говорим, что `String` владеет своими данными, имеется в виду указатель на буфер в куче и метаданные (длина, ёмкость) на стеке.

### Move vs Copy

Для типов, реализующих маркерный трейт `Copy` (`i32`, `f64`, `bool`, `char`, неизменяемые ссылки `&T`, кортежи и массивы из `Copy`), присваивание копирует биты:

```rust
let a = 5;
let b = a;          // a всё ещё валиден
println!("{a} {b}");
```

Для типов, не реализующих `Copy` (`String`, `Vec<T>`, `Box<T>`, `File` и др.), происходит **move**:

```rust
let s1 = String::from("hi");
let s2 = s1;        // s1 перемещён
// println!("{s1}"); // ошибка: value borrowed here after move
println!("{s2}");
```

Что произошло физически: данные в куче не копировались, биты на стеке (ptr+len+cap) были переданы в `s2`, а `s1` помечен как «больше нельзя использовать». Это статическая проверка, в рантайме нет никаких флагов.

Важное правило: тип может быть `Copy` только если все его поля `Copy`. `String` владеет аллокацией, поэтому никогда не может быть `Copy` — иначе после копирования у нас было бы два владельца одной памяти и double-free при дропе.

### Move в функции

Передача в функцию — тоже move (или копирование для `Copy`):

```rust
fn take(s: String) {
    println!("{}", s);
}                       // s тут дропается

fn main() {
    let x = String::from("a");
    take(x);
    // x недоступен дальше — он перемещён в функцию
}
```

Если хочется и передать, и сохранить владение — функция может вернуть значение обратно:

```rust
fn take_and_give_back(s: String) -> String {
    println!("{s}");
    s
}

fn main() {
    let s = String::from("a");
    let s = take_and_give_back(s);   // снова получили владение
    println!("{s}");
}
```

Такой стиль используют редко — обычно функции принимают **ссылки** (`&T`, `&mut T`), о них в уроке 5.

### Clone — явная глубокая копия

Если нужна именно полная копия данных, делают `.clone()`:

```rust
let s1 = String::from("hi");
let s2 = s1.clone();   // глубокая копия буфера в куче
println!("{s1} {s2}");
```

`Clone` всегда явный — никаких «тихих» копирований. Это разделение между `Copy` (дёшево, биты на стеке) и `Clone` (потенциально дорого, может затрагивать кучу) — одна из самых ценных фишек языка.

### Drop и порядок дропа

В конце области видимости значения дропаются в **обратном порядке** объявления:

```rust
struct Logged(&'static str);
impl Drop for Logged {
    fn drop(&mut self) { println!("drop {}", self.0); }
}

fn main() {
    let _a = Logged("a");
    let _b = Logged("b");
    let _c = Logged("c");
}
// Печатает: drop c, drop b, drop a
```

Это критично для RAII-ресурсов: если вы взяли мьютекс `MutexGuard`, потом открыли файл — гард освободится после файла, как обычно и нужно.

Trait `Drop` — это как деструктор в C++. `drop(&mut self)` вызывается автоматически. Реализуют его, когда тип владеет внешним ресурсом (соединение, lock, FFI-handle).

Принудительно вызвать `Drop` нельзя (`x.drop()` запрещено), но есть стандартная функция `drop(x)`, которая забирает значение во владение и тут же его дропает:

```rust
let s = String::from("освободить пораньше");
drop(s);    // дроп прямо здесь
// s недоступен дальше
```

### Замыкания и захват

Замыкание захватывает переменные либо по ссылке, либо по `mut`-ссылке, либо по значению (`move`). По умолчанию выбирается минимально достаточный режим:

```rust
let v = vec![1, 2, 3];
let f = || println!("{:?}", v);    // захват по & (read-only)
f();
println!("{:?}", v);                // v всё ещё доступен

let f = move || println!("{:?}", v);   // захват по значению
f();
// println!("{:?}", v);                  // v перемещён в замыкание
```

`move`-замыкания нужны там, где замыкание переживает текущий стек: в потоках, async-задачах, колбэках.

### Семантика для коллекций

`Vec<T>` владеет элементами:

```rust
let words = vec![String::from("a"), String::from("b")];
for w in words {            // забираем владение каждой строкой
    println!("{w}");
}
// words недоступен — он съеден циклом

let words = vec![String::from("a"), String::from("b")];
for w in &words {           // итерация по ссылкам
    println!("{w}");
}
println!("{words:?}");      // ок, words доступен
```

Это одна из самых частых «обрабатывает ли мой цикл данные или забирает их?» — разница в одном амперсанде.

### Распространённые ошибки и сообщения компилятора

1. **value borrowed here after move** — попытка использовать переменную, ownership которой уже передан. Решение: `.clone()`, либо передавать ссылку.
2. **cannot move out of borrowed content** — попытка сделать move через `&` или `&mut`. Решение: `.clone()`, либо изменить дизайн.
3. **cannot return reference to local variable** — функция пытается вернуть ссылку на локальную переменную. Решение: вернуть значение.
4. **partially moved value** — у структуры один из полей перемещён. Решение: переместить всё структурой целиком, либо клонировать поле.

```rust
    println!("show: {s}");

fn main() {
```

```rust
    println!("{s}");
}
```

### Задача 3. Clone и Copy

```rust
#[derive(Clone)]
struct Data { name: String, version: u32 }

fn use_by_value(d: Data) {
    println!("{} v{}", d.name, d.version);
}

fn main() {
    let d = Data { name: "rustup".into(), version: 1 };
    use_by_value(d.clone());   // клонируем
    use_by_value(d);            // последний раз — отдаём
    // use_by_value(d);          // ошибка: value moved
}
```

### Задача 4. Drop в действии

```rust
struct Res(&'static str);
impl Drop for Res {
    fn drop(&mut self) { println!("освобождён: {}", self.0); }
}

fn main() {
    let _a = Res("a");
    {
        let _b = Res("b");
        println!("внутри блока");
    }   // здесь дроп b
    println!("после блока");
}   // здесь дроп a
```

Запустите и убедитесь в порядке вывода.

### Задача 5. Замыкание с move

```rust
fn make_printer(label: String) -> impl Fn(i32) {
    move |n| println!("{label}: {n}")
}

fn main() {
    let p = make_printer(String::from("счёт"));
    p(1);
    p(2);
    p(3);
}
```

Здесь `move` обязателен: `label` нужно «утащить» внутрь замыкания, потому что замыкание переживёт функцию `make_printer`.

### Задача 6. Поток и move

```rust
use std::thread;

fn main() {
    let v = vec![1, 2, 3, 4];
    let handle = thread::spawn(move || {
        println!("в потоке: {:?}", v);
        v.iter().sum::<i32>()
    });
    let sum = handle.join().unwrap();
    println!("сумма = {sum}");
}
```

Без `move` компилятор не разрешит передать `v` в поток: он требует, чтобы данные пережили `spawn`, а замыкание по умолчанию захватывает только ссылку.

### Задача 7. Partial move в структурах

```rust
struct User { name: String, age: u32 }

fn main() {
    let u = User { name: String::from("Аня"), age: 30 };
    let User { name, age } = u;     // частичная деструктуризация
    println!("{name}, {age}");
    // println!("{}", u.name);       // ошибка: используется после move
}
```

```rust
fn first(v: &Vec<String>) -> String {
```

```rust
impl Drop for R { fn drop(&mut self) { println!("drop {}", self.0); } }
```

---

### Ответы

1. `s1` перемещён в `s2`; использование переменной после move запрещено. `String` не реализует `Copy`, потому что владеет аллокацией на куче.

2. `&T` реализует `Copy` — это просто указатель, безопасно копировать. `&mut T` **не** реализует `Copy`: иначе нарушалось бы правило «одна изменяемая ссылка», и можно было бы получить две `&mut` на одни данные. Это сломало бы инварианты заимствований и открыло бы дверь к гонкам.

3. `move` переместит `v` в замыкание; раскомментированная строка не скомпилируется (`v` уже перемещён). Без `move` замыкание захватило бы по ссылке, и `v` был бы доступен после вызова.

4. В обратном порядке: `c`, `b`, `a`. Это критично для RAII-ресурсов: lock guard, открытый файл, транзакция БД — всё освобождается «снаружи внутрь», как ожидается.

5. Не скомпилируется. После `let t = s;` переменная `s` перемещена и недоступна — `let s = s;` пытается прочитать невалидное значение. Чтобы переиспользовать имя, нужно `let s = t;` или `let s = String::from(...)`.

6. `String` владеет буфером на куче. Если бы у него был `Copy`, при `let s2 = s1` появилось бы два владельца одного буфера — и при дропе обоих был бы double-free. `&str` — это жирный указатель (ptr + len) на чужие данные; он ничем не владеет, копирование безопасно.

7. `[1, 2, 3, 99]`. Вектор `v` перемещён в `take`, там стал `mut`, добавилось `99`, владение вернулось через `return v`.

8. `v[0]` пытается переместить `String` из вектора через ссылку — это запрещено (`cannot move out of borrowed content`). Починить: вернуть ссылку `&String`/`&str` (`-> &str { &v[0] }`), либо клонировать (`v[0].clone()`), либо забирать вектор во владение и сделать `v.into_iter().next().unwrap()`.

9. `std::mem::drop(x)` — обычная функция, она забирает значение во владение и при выходе из себя вызывает реализацию `Drop`. `x.drop()` — это вызов метода трейта `Drop::drop`, и он **запрещён** напрямую: компилятор не позволит вызывать его, потому что после ручного `drop` значение всё ещё «существует» с точки зрения системы типов и было бы дропнуто ещё раз в конце скоупа — двойное освобождение. Поэтому `drop(x)` — единственный способ вручную освободить значение пораньше.

10. Напечатает: `drop x`, потом `середина`, потом `drop y`. `drop(_x)` форсирует дроп `x` сразу. Затем выводится «середина». В конце `main` дропается `_y`.
# Урок 4. Ownership (владение)

## Теория

### Три правила

1. У каждого значения есть один владелец.
2. В каждый момент времени владелец один.
3. Когда владелец выходит из scope — значение дропается (вызывается `Drop`).

Это исключает double-free, use-after-free и data races на этапе компиляции, без сборщика мусора.

### Move vs Copy

Для типов, реализующих `Copy` (целые, `bool`, `char`, `f64`, кортежи из `Copy`), присваивание копирует биты:

```rust
let a = 5;
let b = a; // a всё ещё валиден
```

Для не-`Copy` (`String`, `Vec`, `Box` и пр.) — происходит **move**:

```rust
let s1 = String::from("hi");
let s2 = s1;       // s1 перемещён
// println!("{}", s1); // ошибка компиляции
```

Передача в функцию — тоже move (или копирование для Copy):

```rust
fn take(s: String) { /* ... */ } // забирает владение
let x = String::from("a");
take(x);
// x недоступен дальше
```

### Clone

`.clone()` делает явное глубокое копирование. Это всегда видимо в коде — не происходит "молча".

### Drop

В конце scope значения дропаются в **обратном порядке** объявления. Можно реализовать `impl Drop` для пользовательских освобождений (файлы, lock'и).

## Практика

Напишите функцию с move и без:

```rust
fn take_and_give_back(s: String) -> String {
    println!("внутри: {}", s);
    s
}

fn make_greeting(name: &str) -> String {
    format!("Привет, {}", name)
}

fn main() {
    let s = String::from("мир");
    let s = take_and_give_back(s); // получаем владение обратно
    let g = make_greeting("Алиса"); // не отдаём владение строкой
    println!("{} | {}", s, g);
}
```

## Тест

**1.** Почему этот код не компилируется?
```rust
fn main() {
    let s1 = String::from("a");
    let s2 = s1;
    println!("{}", s1);
}
```

**2.** Реализует ли `&T` трейт `Copy`? А `&mut T`?

**3.** Что произойдёт?
```rust
fn main() {
    let v = vec![1, 2, 3];
    let f = move || println!("{:?}", v);
    f();
    // println!("{:?}", v); // что здесь?
}
```

**4.** В каком порядке дропнутся `a`, `b`, `c`?
```rust
fn main() {
    let a = String::from("a");
    let b = String::from("b");
    let c = String::from("c");
}
```

**5.** Скомпилируется ли?
```rust
fn main() {
    let s = String::from("x");
    let t = s;
    let s = s; // !
}
```

---

### Ответы

1. `s1` перемещён в `s2`; использование после move запрещено. `String` не реализует `Copy`, потому что владеет аллокацией на куче.

2. `&T` реализует `Copy` — это просто указатель, безопасно копировать. `&mut T` **не** реализует `Copy`, иначе нарушалось бы правило "одна изменяемая ссылка".

3. `move` переместит `v` в замыкание; раскомментированная строка не скомпилируется. Без `move` замыкание захватило бы по ссылке, и `v` был бы доступен.

4. В обратном порядке: `c`, `b`, `a`. Это важно для RAII-ресурсов (mutex guard и т.п.).

5. Не скомпилируется: после `let t = s;` переменная `s` перемещена и недоступна. Чтобы переиспользовать имя, нужно `let s = t;` или `let s = String::from(...)`.
