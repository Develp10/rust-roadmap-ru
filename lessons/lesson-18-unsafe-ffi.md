# Урок 18. Unsafe Rust и FFI

`unsafe` — это не «выключатель проверок», а контракт. Вы говорите компилятору: «здесь я беру ответственность за инварианты, которые он не может проверить сам». Этот блок нужен, чтобы взаимодействовать с C, аппаратурой, реализовывать стандартные коллекции, оптимизировать горячий код. И это же — самый частый источник UB, если использовать его неаккуратно.

## Теория

### Что разрешает `unsafe`

В `unsafe`-блоке (или `unsafe fn`) появляются 5 «суперспособностей», запрещённых в безопасном Rust:

1. Разыменовывать «сырые» указатели (`*const T`, `*mut T`).
2. Вызывать `unsafe fn`.
3. Изменять статические переменные (`static mut`).
4. Реализовывать `unsafe` трейты.
5. Обращаться к полям `union`.

Всё остальное проверяется как обычно: borrow checker не отключается.

### Сырые указатели

```rust
let mut x = 10;
let r1: *const i32 = &x;
let r2: *mut i32   = &mut x;

unsafe {
    println!("{}", *r1);
    *r2 = 20;
}
```

Сырые указатели не имеют времени жизни и не блокируют borrow checker. Зато за валидность данных отвечаете вы.

### Контракт безопасной обёртки

Идиома: внутри `unsafe` всё опасное, наружу — безопасный API. Пример (упрощённая `split_at_mut`):

```rust
use std::slice;

pub fn split_at_mut<T>(slice: &mut [T], mid: usize) -> (&mut [T], &mut [T]) {
    let len = slice.len();
    assert!(mid <= len);
    let ptr = slice.as_mut_ptr();
    unsafe {
        (
            slice::from_raw_parts_mut(ptr, mid),
            slice::from_raw_parts_mut(ptr.add(mid), len - mid),
        )
    }
}
```

Здесь `unsafe` оправдан: оба слайса не пересекаются, длины проверены — ни одного UB.

### Undefined Behavior (UB)

Самые частые источники UB:

- Разыменование `null` или висячего указателя.
- Чтение неинициализированной памяти.
- Нарушение алиасинга: одновременная мутабельная и иммутабельная ссылка на одни данные.
- Несоответствие лайфтаймов в `from_raw_parts`.
- Гонки данных в `static mut`.

UB в Rust столь же опасен, как в C: компилятор имеет право предположить, что его не существует, и оптимизировать соответствующе.

### `unsafe fn` и контракты

Если функция требует выполнения инвариантов от вызывающего, её помечают `unsafe fn`. Документируют это блоком `# Safety`:

```rust
/// Возвращает ссылку на элемент без проверки границ.
///
/// # Safety
/// Вызывающий обязан гарантировать, что `index < self.len()`.
pub unsafe fn get_unchecked<T>(slice: &[T], index: usize) -> &T {
    unsafe { &*slice.as_ptr().add(index) }
}
```

### `static mut` и атомики

`static mut` существует, но почти всегда лучше использовать `AtomicU64`, `Mutex` или `OnceLock`. С `static mut` компилятор не защищает от гонок, и любое чтение/запись требует `unsafe`.

### FFI: вызов C из Rust

Допустим, есть C-функция `int square(int)`. Из Rust:

```rust
extern "C" {
    fn square(x: i32) -> i32;
}

fn main() {
    let r = unsafe { square(7) };
    println!("{r}");
}
```

Линкер должен найти символ — обычно через `build.rs` и крейт `cc`, либо через `#[link(name = "...")]`.

### FFI: экспорт Rust в C

```rust
#[no_mangle]
pub extern "C" fn rust_add(a: i32, b: i32) -> i32 {
    a + b
}
```

`#[no_mangle]` запрещает компилятору менять имя символа. `extern "C"` фиксирует ABI вызова. Чтобы получить динамическую библиотеку, в `Cargo.toml`:

```toml
[lib]
crate-type = ["cdylib"]
```

### Строки на границе FFI

Rust-строка — это UTF-8 без `\\0` в конце. C ждёт `*const c_char` с нулевым терминатором. Используйте `std::ffi::CString` и `CStr`:

```rust
use std::ffi::{CStr, CString};
use std::os::raw::c_char;

extern "C" { fn puts(s: *const c_char) -> i32; }

fn main() {
    let cs = CString::new("Привет из Rust").unwrap();
    unsafe { puts(cs.as_ptr()); }
}
```

### Bindings

Писать `extern`-блоки руками для сложных C-API скучно и опасно. Утилита `bindgen` генерирует биндинги по C-заголовкам. Обратно — `cbindgen` создаёт C-заголовки из Rust API.

### Инструменты для проверки

- **Miri** — интерпретатор MIR, который ловит UB в `unsafe`. Запуск: `cargo +nightly miri test`.
- **AddressSanitizer / ThreadSanitizer** — через `-Z sanitizer=address` на nightly.
- **Loom** — модельный чекер для проверки конкурентного кода.
- **Clippy** имеет лекции типа `clippy::undocumented_unsafe_blocks`.

Правило: на каждый `unsafe`-блок пишем комментарий `// SAFETY: ...`, поясняющий, почему инварианты выполнены.

## Практика

Завернём C-функцию в безопасный Rust-API. Пусть библиотека `mylib` экспортирует функцию `int sum(const int* arr, size_t len)`.

Файл `build.rs`:

```rust
fn main() {
    cc::Build::new().file("c/sum.c").compile("mylib");
}
```

Файл `c/sum.c`:

```c
#include <stddef.h>
int sum(const int* arr, size_t len) {
    int s = 0;
    for (size_t i = 0; i < len; i++) s += arr[i];
    return s;
}
```

Файл `src/lib.rs`:

```rust
use std::os::raw::{c_int};

extern "C" {
    fn sum(arr: *const c_int, len: usize) -> c_int;
}

/// Сумма элементов массива через C-функцию.
pub fn safe_sum(values: &[i32]) -> i32 {
    // SAFETY: указатель и длина из живого слайса валидны на всё время вызова,
    // C-функция не сохраняет указатель и не пишет в него.
    unsafe { sum(values.as_ptr(), values.len()) }
}

#[cfg(test)]
mod tests {
    use super::*;
    #[test]
    fn empty_is_zero() { assert_eq!(safe_sum(&[]), 0); }
    #[test]
    fn small() { assert_eq!(safe_sum(&[1, 2, 3, 4]), 10); }
}
```

Здесь весь `unsafe` спрятан внутри `safe_sum`, и пользователь библиотеки не знает о существовании C — для него это обычная функция.

## Тест

1. Что именно «разрешает» делать ключевое слово `unsafe`, и что оно при этом *не отключает*?
2. Какие 3 классических причины UB в `unsafe`-коде вы запомнили?
3. Зачем функцию помечают `unsafe fn` вместо использования `unsafe` блока внутри?
4. Какие типы используются для передачи строк через FFI и почему `String` нельзя отдать напрямую в C?
5. Что такое Miri и какую ошибку он поможет поймать раньше, чем продакшен?

---

### Ответы

1. `unsafe` разрешает: разыменовывать сырые указатели, вызывать `unsafe fn`, обращаться к `static mut`, реализовывать unsafe-трейты и читать поля `union`. Он *не* отключает borrow checker, проверки типов или владение — эти правила продолжают работать.
2. Например: разыменование висячего/null указателя; чтение неинициализированной памяти; одновременное существование `&mut` и любых других ссылок (нарушение алиасинга). Также частые: гонки данных в `static mut` и неверные лайфтаймы в `from_raw_parts`.
3. `unsafe fn` означает, что вызов функции требует от вызывающего выполнения инвариантов — компилятор не может их проверить. Это «инфекционный» маркер: каждый, кто её вызывает, обязан понимать и гарантировать контракт. Если же `unsafe` нужен только для деталей реализации, а API безопасен, маркер на функции ставить не надо.
4. Используют `std::ffi::CString` (владение) и `CStr` (заём). `String` хранит UTF-8 без нулевого терминатора и может содержать внутренние `\\0`. C-функции ждут указатель на `\\0`-терминированную строку, поэтому передача `String` напрямую — ошибка дизайна и потенциальный UB.
5. Miri — интерпретатор MIR (промежуточного представления Rust), который при запуске тестов проверяет код на UB: разыменование висячих указателей, нарушения алиасинга, выходы за границы и т.д. Он ловит классы ошибок, которые могут не проявиться в обычной сборке, но «выстрелят» при смене компилятора или платформы.
