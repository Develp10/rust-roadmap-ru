# Урок 16. Тестирование

Хорошие тесты — это страховка от регрессий и инструмент проектирования. Rust встроил тестирование в `cargo`: тесты запускаются командой `cargo test`, не требуют отдельных фреймворков и параллельно работают по умолчанию.

## Теория

### Виды тестов

В Rust различают три типа тестов:

1. **Unit-тесты** — лежат в том же файле, что и тестируемый код, в модуле `#[cfg(test)] mod tests`. Имеют доступ к приватным элементам.
2. **Integration-тесты** — отдельные файлы в `tests/` рядом с `src/`. Видят только публичный API крейта.
3. **Doc-тесты** — примеры в комментариях `///`, которые компилируются и запускаются как тесты.

### Простейший тест

```rust
pub fn add(a: i32, b: i32) -> i32 { a + b }

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn addition_works() {
        assert_eq!(add(2, 3), 5);
    }
}
```

`#[cfg(test)]` означает, что модуль компилируется только при `cargo test`, и не попадает в обычный билд.

### Макросы для проверок

- `assert!(условие)` — паника, если `условие == false`.
- `assert_eq!(left, right)` — проверка равенства, печатает оба значения при провале.
- `assert_ne!(left, right)` — проверка неравенства.
- `assert!(cond, "сообщение {:?}", val)` — пользовательское сообщение.

### Тесты, которые должны паниковать

```rust
#[test]
#[should_panic(expected = "деление на ноль")]
fn divide_by_zero_panics() {
    divide(10, 0);
}
```

`expected` — подстрока, которую должна содержать паника. Без неё тест считается успешным при любой панике, что иногда маскирует ошибку.

### Возврат `Result` из теста

Удобно, когда внутри теста много операций `?`:

```rust
#[test]
fn parses_config() -> Result<(), Box<dyn std::error::Error>> {
    let cfg: Config = serde_json::from_str(r#"{"name":"test"}"#)?;
    assert_eq!(cfg.name, "test");
    Ok(())
}
```

Тест считается провальным, если возвращён `Err`.

### `#[ignore]` и фильтры

Тесты с `#[ignore]` пропускаются по умолчанию и запускаются командой `cargo test -- --ignored`. Полезно для долгих интеграционных проверок.

Запуск отдельных тестов:

```bash
cargo test addition          # все тесты, в имени которых есть "addition"
cargo test tests::addition_works
cargo test -- --nocapture    # не подавлять stdout
cargo test -- --test-threads=1
```

### Интеграционные тесты

Структура проекта:

```
my_crate/
├── src/
│   └── lib.rs
└── tests/
    ├── api.rs
    └── common/
        └── mod.rs
```

Файл `tests/api.rs`:

```rust
use my_crate::add;

#[test]
fn integration_add() {
    assert_eq!(add(40, 2), 42);
}
```

Каждый файл в `tests/` собирается как отдельный крейт. Общий код вспомогательных функций кладут в `tests/common/mod.rs` (именно через подмодуль, иначе `common.rs` сам станет тестом).

### Doc-тесты

```rust
/// Складывает два числа.
///
/// # Примеры
///
/// ```
/// use my_crate::add;
/// assert_eq!(add(2, 2), 4);
/// ```
pub fn add(a: i32, b: i32) -> i32 { a + b }
```

`cargo test` запустит этот пример. Это лучшая защита документации от устаревания.

### Property-based тесты

Помимо обычных тестов, есть стиль *property-based*: проверяется свойство для случайных входов. Популярные крейты — `proptest` и `quickcheck`.

```rust
use proptest::prelude::*;

proptest! {
    #[test]
    fn addition_is_commutative(a: i32, b: i32) {
        prop_assert_eq!(a.wrapping_add(b), b.wrapping_add(a));
    }
}
```

### Бенчмарки и измерения

Стабильный Rust не имеет встроенного бенчмаркера, но крейт `criterion` — индустриальный стандарт. Он считает статистику, строит графики и обнаруживает регрессии производительности.

### Покрытие и mocking

- `cargo llvm-cov` или `tarpaulin` — покрытие кода.
- `mockall` — генерация mock-объектов для трейтов.
- `insta` — snapshot-тестирование (сравнение со «слепком» вывода).

## Практика

Создадим библиотеку `calc` с unit- и интеграционными тестами:

Файл `src/lib.rs`:

```rust
#[derive(Debug, PartialEq)]
pub enum CalcError {
    DivByZero,
}

pub fn divide(a: i64, b: i64) -> Result<i64, CalcError> {
    if b == 0 {
        Err(CalcError::DivByZero)
    } else {
        Ok(a / b)
    }
}

/// Возвращает n-й член последовательности Фибоначчи.
///
/// # Примеры
///
/// ```
/// assert_eq!(calc::fib(10), 55);
/// ```
pub fn fib(n: u32) -> u64 {
    let (mut a, mut b) = (0u64, 1u64);
    for _ in 0..n {
        let next = a + b;
        a = b;
        b = next;
    }
    a
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn divide_normal() {
        assert_eq!(divide(10, 2), Ok(5));
    }

    #[test]
    fn divide_zero() {
        assert_eq!(divide(1, 0), Err(CalcError::DivByZero));
    }

    #[test]
    fn fib_small() {
        for (i, expected) in [(0, 0), (1, 1), (2, 1), (3, 2), (10, 55)] {
            assert_eq!(fib(i), expected, "fib({i})");
        }
    }
}
```

Файл `tests/integration.rs`:

```rust
use calc::{divide, fib, CalcError};

#[test]
fn smoke() {
    assert_eq!(divide(100, 5), Ok(20));
    assert_eq!(divide(1, 0), Err(CalcError::DivByZero));
    assert_eq!(fib(20), 6765);
}
```

Запускаем:

```bash
cargo test
cargo test fib
cargo test --doc
```

## Тест

1. Чем unit-тесты отличаются от интеграционных по расположению и видимости?
2. Что делает атрибут `#[cfg(test)]`?
3. Зачем нужен `expected = "..."` в `#[should_panic]`?
4. Почему файл `tests/common.rs` — плохая идея, и как сделать правильно?
5. Что такое doc-тесты и как они помогают поддерживать документацию?

---

### Ответы

1. Unit-тесты лежат рядом с кодом в модуле `#[cfg(test)] mod tests` и видят приватные функции. Интеграционные — отдельные файлы в каталоге `tests/` в корне проекта; они работают только с публичным API, как внешние пользователи крейта.
2. `#[cfg(test)]` включает компиляцию модуля или элемента только под флагом тестирования (`cargo test`). В обычном билде такой код не попадает в бинарник.
3. `expected` фиксирует, какая именно паника ожидалась. Без него тест проходит при любой панике, в том числе посторонней (например, `unwrap` в чужом месте), что маскирует баги.
4. Каждый `.rs`-файл в `tests/` собирается как отдельный крейт и запускает тесты. `tests/common.rs` тоже стал бы таким крейтом, но без тестов и с шумом в выводе. Правильно — `tests/common/mod.rs`: он считается подмодулем и не запускается самостоятельно.
5. Doc-тесты — это примеры кода в комментариях `///` между тройными бэктиками. `cargo test` компилирует и запускает их, поэтому если код в документации устареет, сборка упадёт. Это защищает документацию от рассинхрона с реальностью.
